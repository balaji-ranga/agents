# Agent Intelligence Knowledge Graph

Implementation plan for a Neo4j (or compatible open-source graph database) knowledge graph that captures internal OpenClaw agent behavior, workflow execution, Kanban feedback, and drives autonomous self-improvement with a progressive dashboard.

**Scope:** Internal OpenClaw agents only (`agents.openclaw_agent_id` set). External registered agents (`external_agents`) are scored on penalties only — no self-improvement loop.

---

## 1. Choose graph store and wire it into deploy

- Add **Neo4j Community** (or **Memgraph** / **FalkorDB** for lighter ops) to `deploy/docker-compose.yml` with a dedicated volume and internal network only.
- Add env vars: `GRAPH_DB_URI`, `GRAPH_DB_USER`, `GRAPH_DB_PASSWORD`, `GRAPH_ENABLED=1`.
- Create `backend/src/services/graph-db.js` — thin driver wrapper with `mergeNode`, `mergeEdge`, `runCypher`, health check.

---

## 2. Define the graph ontology (schema-first)

### Node labels

| Label | Source in Agent OS |
|---|---|
| `Agent` | `agents` where `openclaw_agent_id` is set (internal only) |
| `ExternalAgent` | `external_agents` |
| `WorkflowDefinition` | `agent_workflow_definitions` |
| `WorkflowNode` | nodes inside `published_graph_json` |
| `WorkflowRun` | `agent_workflow_runs` |
| `StepExecution` | `agent_workflow_run_steps` |
| `KanbanTask` | `kanban_tasks` |
| `Delegation` | `agent_delegation_tasks` |
| `Feedback` | synthesized from rejections, reopens, comments |
| `ToolInvocation` | `content_tool_logs`, `mcp_call_logs` |
| `Improvement` | generated by evaluator agent |
| `ScoreSnapshot` | periodic agent/workflow scores |

### Relationship types

- `(:Agent)-[:EXECUTED]->(:StepExecution)`
- `(:WorkflowRun)-[:HAS_STEP]->(:StepExecution)`
- `(:StepExecution)-[:PRECEDED_BY]->(:StepExecution)` (branch handle on edge)
- `(:StepExecution)-[:PRODUCED]->(:Feedback)`
- `(:KanbanTask)-[:LINKED_TO]->(:StepExecution)` via `parseAgentWorkflowMeta()` in task description
- `(:KanbanTask)-[:ASSIGNED_TO]->(:Agent)`
- `(:Feedback)-[:CAUSED_BY]->(:Agent|:ExternalAgent)`
- `(:Agent)-[:RECEIVED_IMPROVEMENT]->(:Improvement)`
- `(:Improvement)-[:ADDRESSES]->(:Feedback|:StepExecution)`

Store `source_table`, `source_id`, `tenant_id`, `timestamp` on every node/edge for traceability back to SQLite.

## 3. Build an event emitter that mirrors SQLite writes to the graph

Create `backend/src/services/graph-event-emitter.js` with a single `emitGraphEvent(eventType, payload)` API. Hook it at these existing write points (non-blocking, fire-and-forget with retry queue):

| Hook location | Events captured |
|---|---|
| `agent-workflow-runner.js` → `upsertStep()` | step start/complete/fail, node I/O, branch taken |
| `agent-workflow-run-manager.js` | run started/paused/completed/failed |
| `agent-workflow-store.js` → `appendAudit()` | publish, CEO approve/reject, graph diffs |
| `agent-workflow-kanban.js` | Kanban task created/linked to workflow step |
| `routes/kanban.js` | status change, reopen, new `task_messages` |
| `delegation-queue.js` | delegation prompt → response/error |
| `routes/agents.js` | `chat_turns`, standalone agent actions |
| `content_tool_logs` insert | tool calls (especially `kanban_move_status`) |
| `mcp_call_logs` insert | MCP tool loop from brain nodes |
| `agent-workflow-brain.js` | `toolCallLog` from brain output → ToolInvocation nodes |

Use an **outbox table** in SQLite (`graph_event_outbox`) so graph writes survive Neo4j downtime and can be replayed.

---

## 4. Classify feedback into graph edges (the "understanding" layer)

On every human or system signal, create a `Feedback` node with a typed label:

| Signal | Type | Graph action |
|---|---|---|
| CEO approval reject + comment | `REJECTION` | `Feedback` → `CAUSED_BY` → responsible `Agent`/`StepExecution` |
| Kanban reopen | `REOPEN` | link to prior `StepExecution`, capture comment thread |
| `task_messages` (user role) | `CORRECTION` | link to assigned agent + step |
| Step `status=failed` / `error_message` | `FAILURE` | auto-link to executing agent |
| Delegation `error_message` | `DELEGATION_FAILURE` | link to `Delegation` → `Agent` |
| External agent bad outcome | `EXTERNAL_LOW_QUALITY` | `CAUSED_BY` → `ExternalAgent` |

Parse CEO approval output (`{ decision: 'rejected', comment }`) and Kanban metadata tags (`[agent_workflow:nodeId]`, `agent_wf_run_id`) to join feedback to the exact workflow step.

## 6. Define scoring model (internal vs external agents)

Create `backend/src/services/agent-performance-scorer.js`:

### Internal OpenClaw agent score (0–100, rolling 30-day window)

- +10 completed step without rework
- −25 CEO rejection on linked step
- −15 Kanban reopen within 7 days of completion
- −10 user correction message after agent response
- −20 step failure / delegation error
- +5 successful tool invocation without retry

### External agent score (separate, lower weight)

- Any linked rejection/failure/reopen where `ExternalAgent` appears in the causal chain → −30 per incident
- No positive scoring path for external agents (penalty-only)

Persist `ScoreSnapshot` nodes weekly; keep latest score on `Agent` node property `current_score`.

---

## 7. Build the autonomous evaluator (self-improvement engine)

Create `backend/src/services/agent-self-improver.js` — a scheduled job (cron or workflow trigger) that:

1. Runs Cypher queries to find patterns: repeated rejections on same node type, same agent, same tool, same prompt pattern.
2. Clusters related `Feedback` nodes (by agent + node_type + keyword in comment).
3. Invokes an **internal OpenClaw evaluator agent** (e.g. COO or dedicated `performance-reviewer` agent) with context: failing steps, feedback text, prior improvements, current SOUL.md/MEMORY.md snippets.
4. Evaluator outputs structured JSON:

```json
{
  "improvement_type": "memory_update|prompt_patch|tool_grant|workflow_suggestion",
  "target_agent_id": "...",
  "rationale": "...",
  "patch": "...",
  "confidence": 0.85
}
```

5. Creates `Improvement` node linked to addressed feedback.
6. Applies low-risk changes automatically (append to `MEMORY.md`, update workspace TOOLS.md guidance); queues high-risk changes (workflow graph edits, tool grant changes) for CEO approval via Kanban card.

### Evaluator triggers

- Nightly cron
- After every 5th rejection/reopen for same agent
- On workflow run failure streak ≥ 3

## 8. Close the loop — inject improvements back into agents

| Improvement type | Apply via |
|---|---|
| Memory/guidance | Write to agent workspace `MEMORY.md` / `SOUL.md` via existing workspace file APIs |
| Tool usage | Update via `openclaw-agent-tools.js` grant sync |
| Workflow fix | Create draft in `agent_workflow_definitions.draft_graph_json`, notify via Kanban |
| Prompt tuning | Update brain node config in workflow definition |

After applying, mark `Improvement` node `status: applied` and link to a follow-up `ScoreSnapshot` to measure delta.

New page: `frontend/src/pages/AgentIntelligence.jsx` at `/intelligence`.

### Widgets

1. **Score trend line** — per internal agent, 30/90-day rolling score (Recharts line chart)
2. **Improvement timeline** — applied improvements with before/after score delta
3. **Feedback heatmap** — rejections/reopens by workflow node type and agent
4. **Live graph explorer** — React Flow or Neo4j NVL embedded view showing agent → step → feedback → improvement chain for selected run/agent
5. **External agent penalty panel** — registered external agents ranked by incident count (read-only, no self-improvement)

Reuse patterns from `AgentWorkflows.jsx` (run inspector, step tooltips) and `@xyflow/react` already in the project.

## 11. Scope guard — internal OpenClaw agents only

- In graph emitter: only create `Agent` nodes where `agents.openclaw_agent_id IS NOT NULL` and `agent_type != 'external'`.
- Brain nodes and delegation nodes resolve to internal agent IDs only.
- External agents get `ExternalAgent` label; their actions create penalty edges but never trigger self-improvement on themselves.
- Filter evaluator queries: `WHERE agent.openclaw_agent_id IS NOT NULL`.

---

## 12. Observability and safety rails

- Log all auto-applied improvements to `agent_workflow_audit` (mirror pattern).
- CEO approval gate for: workflow graph changes, tool grant revocations, SOUL.md rewrites > N chars.
- Rate-limit auto MEMORY.md updates (max 1 per agent per day).
- Dashboard shows "pending approval" improvements separately from "auto-applied".
- Add `GRAPH_ENABLED=0` kill switch; SQLite remains source of truth.

## 15. Success criteria

- Every workflow run with a brain/agent node produces a connected subgraph in Neo4j.
- A CEO rejection on a Kanban-linked approval step creates a `Feedback:REJECTION` node within seconds.
- Nightly evaluator produces at least one `Improvement` when rejection rate > threshold.
- Dashboard shows score rising after applied improvements on the same agent over 2+ weeks.
- External agent linked to a failed delegation shows score drop with no auto-improvement applied to it.

---

## Graph DB choice

**Neo4j Community** is the recommended default — Cypher for pattern matching (repeated rejection chains), good visualization tooling, and straightforward Docker setup. **Memgraph** is a valid alternative for lower memory footprint if the team already knows openCypher.
